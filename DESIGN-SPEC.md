# SLE · LOG 产品设计稿

## 1. 产品定位

面向需要长期记录 SLE 化验和治疗过程的个人用户。产品只负责记录、整理和可视化，不生成诊断结论，不提供药物调整建议。

设计原则：

- 准确优先：保存当次单位、参考范围、品牌、剂量和频率快照。
- 历史可追溯：调整用药时创建新阶段，不覆盖旧阶段。
- 克制表达：黑白层级为主，语义颜色只表示范围和错误状态。
- 录入友好：移动端保证 44px 点击区域，批量检查按四个分类切换。
- 因果中立：指标和用药仅按时间对照，不推断因果关系。

## 2. 视觉系统

- 设计语言：PandaAI Mono
- 中文字体：Noto Sans SC
- 拉丁字符与数值：Inter
- 技术标识：IBM Plex Mono
- 页面背景：`#F0F0F0`
- 主表面：`#FFFFFF`
- 主文字和主要操作：`#111111`
- 状态色：绿色表示范围内，橙色表示偏高或偏低，红色表示错误或破坏性操作
- 容器圆角：14-16px
- 控件圆角：8-10px
- 动效：160-320ms，只解释选择、切换和保存结果

## 4. 核心流程

### 新增检查

1. 填写检查日期和医院。
2. 在四个分类 Tab 中填写本次实际检查的结果。
3. 未填写结果的项目不会保存。
4. 保存后更新最新结果、趋势图和历史记录。
5. 每条记录保存当次单位和参考范围。

### 用药阶段调整

1. 从当前用药选择“调整剂量”。
2. 填写调整日期、调整类型、新剂量、频率和原因。
3. 结束上一阶段并创建新阶段。
4. 药物详情同时保留所有阶段。

### 单抗实际给药

1. 先创建治疗方案。
2. 每次实施后单独记录日期、剂量、地点、完成状态和不良反应。
3. 系统不会根据计划日期自动标记为完成。

### 数据恢复

1. 导出 JSON 完整备份。
2. 从“我的 - 数据管理”选择备份文件。
3. 校验文件结构。
4. 重新载入恢复后的指标、用药和账号设置。

## 5. 状态设计

- 加载：按钮保持宽度并显示旋转状态，页面结构不跳动。
- 空数据：保留内容区域，解释如何添加第一条记录。
- 表单错误：在表单内部显示明确原因。
- 删除确认：使用独立确认对话框，说明影响范围。
- 同步失败：保留重新同步入口，不隐藏本地数据。
- 单位不一致：趋势图上方显示提示，不静默混合比较。

## 6. 响应式规则

- 桌面端：左侧固定一级导航，趋势图和时间轴使用主要宽度。
- 768px 以下：使用底部导航，页面边距为 16px。
- 指标分类和指标名称支持横向滚动。
- 历史表格转换为标签化记录卡片。
- 用药卡片单列显示，时间轴允许横向滚动。
- 弹窗宽度使用 `calc(100vw - 32px)`，内部独立滚动。
- 日期与下拉控件采用混合交互：桌面端使用统一的自定义月历和选项面板，移动端保留系统原生选择器，并提供 48px 触控区域与清晰的当前值反馈。
- 长选项列表在桌面端支持搜索，选择后即时同步到底层表单字段。

## 7. 1.0 数据边界

- 身份认证：Supabase Auth，用户界面使用用户名与密码登录；内部登录标识不向用户展示。
- 主数据库：Supabase PostgreSQL `public.user_app_state`，每位用户一行 JSONB 健康档案。
- 数据隔离：RLS 按 `auth.uid() = user_id` 限制所有读写。
- 本地缓存：仅用于断网或读取失败时保留当前用户数据，不保存账号密码。
- 注销：调用受限的 `delete_current_user()` RPC，Auth 用户删除后由外键级联删除资料和健康档案。
- 备份：JSON 与 CSV 均不包含登录密码。

正式发布还需要 HTTPS、真实 Supabase 项目联调、账号找回策略、隐私政策、服务协议、数据保留规则和医疗数据合规审查。由于账号不绑定手机号或邮箱，1.0 不提供自助找回密码，需要预先约定管理员协助处理流程。
